# Product Pick Blurb

## Purpose

Every Amazon-affiliate article generated by `SEOArticleAgent` renders an "Our Top Picks" card with up to 5 products. Each pick carries an editorial blurb (`PickReasoning.reasoning` — see `src/pipeline/html-builder.ts`) written by the Kimi K2.5 writer. This skill documents the blurb's shape so the prompt in `src/pipeline/writer.ts` and the rendered output stay in sync.

## Blurb structure (3 sentences, 55–85 words)

Every blurb is three sentences, in this order:

1. **Feature sentence** — one concrete, named attribute of this specific product. Not generic praise. If the product has no verifiable feature in the grounding data, name a category trait (material, capacity, format) instead of inventing one.
2. **Tradeoff + fit sentence** — an honest limitation and the buyer who tolerates it. Example: _"The tradeoff is the proprietary tray cost, which suits owners who value time over supply spend."_
3. **Closing line** — MUST begin with the literal prefix `Why we like this pick:` and follow the template below.

## Closing-line template

```
Why we like this pick: [problem it solves] → [key benefit] → ideal for [buyer or use case].
```

Rules for the closing line:

- **One sentence, 15–30 words.**
- The three beats must all appear. Literal arrow characters (`→`) or natural connective prose (`Solves X, delivers Y, ideal for Z`) are both acceptable — pick one style and stay consistent inside a single article.
- The problem must be phrased from the cat owner's perspective, not the product's. ("Solves daily scooping drudgery" — not "automates scooping.")
- The ideal-for clause names a buyer type or use case, not a restatement of the feature. ("Ideal for busy single-cat owners who travel" — not "Ideal for people who want an auto-raking box.")
- Do not repeat sentences 1 or 2. The closing line reframes the product in buyer-decision terms; it's a summary, not a recap.

## Voice rules

- Plain English. No AI-slop vocabulary: `delve`, `leverage`, `utilize`, `robust`, `comprehensive`, `pivotal`, `cutting-edge`, `game-changer`, `revolutionize`. This list mirrors the ban list already in `buildArticlePrompt()`.
- No em dashes, no exclamation points.
- Do not restate the product name in the closing line — the HTML already renders the product name above the blurb.
- No keyword stuffing. The article's target keyword should not appear inside any `pickReasons` entry.

## Failure modes and detection

- **Model omits the closing line.** The renderer has no enforcement for this — the pick still publishes with whatever the model returned. Detect drift by grep-counting `Why we like this pick:` in the published HTML: expect ≈ `picks × articles`.
- **Model repeats info across sentences.** Observable in a spot-check. The prompt and this skill both explicitly forbid restating.
- **Closing line truncated.** `normalizePickReasons()` in `src/pipeline/writer.ts` caps `reasoning` at 500 characters, which accommodates the 55–85 word budget with headroom. If the model consistently emits longer blurbs and clipping becomes visible, raise the cap again rather than tightening the word budget.

## Integration point

- Worker prompt: `src/pipeline/writer.ts` — `buildArticlePrompt()` builds the per-pick JSON schema string around line 2241. That string must restate the three-sentence structure and the closing-line template so the model has the rules in-context. When this skill changes, update that prompt in the same commit.
- Rendering: `src/pipeline/html-builder.ts` lines 326–417 render `pickReasons` inside `<div class="top-picks">`. The closing line renders inline inside `.pick-reasoning-text` — no CSS or template change needed to ship this skill.
- Validation: `normalizePickReasons()` (same file, lines ~2373–2393) trims and length-caps the field. No new fields are introduced by this skill.
